# v0.0.16 Self-Improvement Program — Dogfooding Report

**Thesis under test:** use praxec's own CPM planner + cohort build-loop to build praxec's improvements, with *commodity* models doing the work via precise micro-waterfall TDD steps, driven by a capable model that delegates rather than hand-codes. When a commodity model can't drive a step, the fix is *more decomposition*, not a more capable model.

This report is the honest A/B + findings synthesis (P8). It is deliberately candid about what worked, what didn't, and what that tells the praxec team.

## What shipped

23 deliverables across the program (mcp-flowgate engine + the cognitive-architectures pack + cpm-planner). Highlights:

- **Durable CPM control plane** (cpm-planner): SQLite persistence + a retry **circuit-breaker** (poison deliverable auto-fails after 3 leases instead of looping forever).
- **INCOSE/SEBoK Vee flow** (`flow.sebok`) beside the SDLC flow, closing its two gaps (first-class Stakeholder Needs; Validation distinct from Verification), adversarially vetted + FMECA'd.
- **Engine hardening**: agent-accessible config reload + lazy-TTL staleness recheck (WSL-safe), deterministic-repair rung, bounded reasoning-stall, per-model **cooldown circuit-breaker** over the chain-walk, fail-fast **credential preflight** + `praxec doctor`, self-provision detection, cost/affinity **telemetry**.
- **Build-loop throughput**: staged `cargo_scope` verdict (scoped inner loop + `--workspace` gate before mark).

## The A/B, honestly

The intended experiment — *existing SDLC loop (A) vs the honed/decomposed loop (B)* on comparable deliverables — did **not** produce a clean controlled result, because a single systemic factor dominated both arms:

### Finding 1 (headline): the build-loop `final_answer` ceremony fails the whole model chain

On a real code deliverable (P10 self-provision), the cohort build-loop's agent **repeatedly produced correct, tested, green code** (a 104-line module with 4 inline tests, `cargo test -p praxec` all-green in **0.85s**) but **could not emit a conforming `final_answer`** → `AGENT_NO_RESULT`. This happened:
- across the **entire model chain** (deepseek-v4-pro → escalated to qwen3-235b-thinking, both fumbled the sign-off),
- on **both** the slow `--workspace` loop **and** the fast scoped loop — so it is **not** a build-speed or budget problem,
- **repeatably** — the model does the creative work, then fails the structured bookkeeping call, and the verified work is discarded.

**Root:** the `final_answer` output contract (done + files_written + commit_message + test_files_written + …) is a failure surface. The more required fields, the more likely a model does the work but never signs off. This is the single biggest blocker to autonomous completion, and it is **model-independent** (the runner itself documents this failure mode).

**Recommended fix (high leverage):** make the sign-off forgiving — harvest `files_written` from `git diff --name-only` when the agent doesn't emit it, and shrink the required-field set to the minimum. A slice with correct code + passing tests must not be thrown away for a missing bookkeeping call. This one change would let the loop actually *complete* code deliverables it already knows how to *write*.

### Finding 2: "one cargo at a time" was a misdiagnosis; scoping works

The rule that serialized all drives was over-generalized. Cargo's build lock is **per target dir**, and each git worktree has its own `target/` — so N parallel worktree drives don't contend. The earlier "runaway" was a *control-plane* loop (multiple procs retrying the same deliverable), now bounded by the cpm-planner circuit-breaker. **Parallel drives are safe** (cap N≈2 for RAM). And **scoping the per-slice cargo to the deliverable's crate** (`-p praxec`) cut a measure step from minutes to **0.85s** — TDD-preserving, with a full `--workspace` gate retained before mark. (Design: `build-throughput-optimization.md`.)

### Finding 3: guard-routed failures silently reported success (pack-wide)

The engine derives `Succeeded` for a terminal reached with no `outcome:` marker. **Every** flow in the pack routed gate aborts to a `failed` terminal via a guard but **none** carried `outcome: failure` — so a flow that aborted reported *success*, and a driver would proceed on a dead workflow. Fixed pack-wide (surfaced by the sebok adversarial vet).

## Telemetry (what the cost report now shows)

P14 makes cost/tokens/**duration**/**affinity** aggregate cleanly from real `agent.completed` + `agent.invoked` audit events, honestly (recorded cost → catalog reprice → uncatalogued `$0`, never fabricated). Example (a 3-step run):

```
  wall time: 3.0s
By affinity:
  reasoning   1 run(s)  $8.0000  1.0s
  coding      1 run(s)  $4.0000  1.0s
```

This is the substrate for a real A/B once Finding 1 is fixed and the loop can complete deliverables unattended.

## Upstream recommendations (for the praxec team) — de-solutioned to requirements

1. **Loop must not discard verified work for a missing sign-off** (Finding 1). *The* completion blocker. Harvest-fallback + minimal required fields.
2. **External drive-progress observability.** From outside a drive you cannot distinguish "waiting on a slow model call" (benign 0 CPU) from "hung." A per-step progress heartbeat is the open gap (the per-call stall-timeout half already shipped in 0.0.15).
3. **Auxiliary-tool state must be repo/run-scoped, not CWD-relative.** `structureos`/`intentos`/`corpus`/`serve` leak `.frontrails/`, `.corpus/`, `.write.lock` into the caller's tree.
4. **Confirmed shipped in 0.0.15 (their wins):** reasoning-stall converted from unbounded silent hang to bounded + classified + escalatable outcome; agent-accessible reload; XDG key resolution; fail-soft serve.

## Bottom line

The thesis holds *for authorship*: commodity + reasoning models, given decomposed steps, reliably **write** correct, tested code. It does **not** yet hold for **completion**, because the loop's structured sign-off is a model-independent failure surface. Fix that one contract and the autonomous loop closes. Everything else the program built — durable planning, the circuit-breakers, the throughput fix, the preflight, the telemetry — is the scaffolding that becomes fully load-bearing the moment the loop can finish what it starts.
